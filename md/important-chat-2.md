## التصور المعماري النهائي المبدئي لمنصة الشؤون التعليمية

### 1. تعريف المنصة

المنصة ليست مجرد “ملف طالب” أو “لوحة إدارة”، بل هي:

```txt
منصة إدارة وتشغيل يومي للمؤسسات التعليمية
```

تخدم نوعين رئيسيين من العمل:

```txt
1. الإدارة والإعداد والتقارير
2. التشغيل اليومي حسب الدور والصلاحية والإسناد
```

لذلك لا نضع كل شيء داخل `web-admin`.

---

## 2. التطبيقات الأساسية

### أ) `web-admin`

هذا مخصص للإدارة الشاملة.

وظائفه:

```txt
- إعداد المؤسسة
- إدارة المدارس
- إدارة السنوات الدراسية
- إدارة الصفوف والفصول
- إدارة الطلاب
- إدارة أولياء الأمور
- إدارة الأشخاص والموظفين
- إدارة العضويات والصلاحيات
- إدارة الإسنادات
- إدارة القوالب والسياسات
- قراءة التقارير
- ملف الطالب الكامل
- ملف الموظف الكامل
- المراجعة الإدارية
```

فكرته:

```txt
web-admin = إعداد + إدارة + تقارير + ملفات شاملة
```

---

### ب) `web-staff`

هذا مخصص للتشغيل اليومي.

لا نحصره في المعلم أو الوكيل أو المشرف فقط، بل يشمل كل دور تشغيلي في المدارس المختلفة، مثل:

```txt
- معلم
- معلمة
- وكيل شؤون الطلاب
- وكيل شؤون المعلمين
- مدير
- مديرة
- مشرف
- مشرفة
- موجه / مرشد طلابي
- معلمة قيم
- معلمة أركان
- معلمة قرآن
- مشرف باص
- أي دور تشغيلي جديد يظهر لاحقًا
```

فكرته:

```txt
web-staff = مهامي اليومية حسب دوري وإسنادي
```

أمثلة مما يظهر للمستخدم حسب صلاحياته:

```txt
- فصولي
- طلابي
- حضور اليوم
- قياسات الفصل
- متابعات الطلاب
- ملاحظات الطلاب
- القضايا والإحالات
- التحفيز
- النقل والباص
- تقييماتي المطلوبة
- المعلمون أو الموظفون المسندون لي
```

---

### ج) `mobile-parent`

يكون لاحقًا لولي الأمر.

وظائفه المستقبلية:

```txt
- متابعة الطالب
- الإشعارات
- الحضور والغياب
- الملاحظات المسموحة
- القضايا المسموح بإظهارها
- الرسائل
```

---

### د) `mobile-staff`

ليس أولوية الآن، لكنه لاحقًا نسخة خفيفة من `web-staff`.

---

## 3. packages/contracts

هذا مصدر الحقيقة للبيانات.

وظيفته:

```txt
- Schemas
- Enums
- Types
- العقود المشتركة بين التطبيقات
```

لا نضع فيه منطق حساب أو صلاحيات معقدة.
هو يعرّف شكل البيانات فقط.

أمثلة:

```txt
Student
Person
Membership
TeacherAssignment
StudentAssessmentRecord
StudentTrackerEntry
StudentLearningLossPlan
EvaluationSubmission
StudentCase
AttendanceRecord
TransportAttendanceRecord
```

---

## 4. packages/domain

هذا لا نؤجله.

يكون من البداية جزءًا أساسيًا، حتى لا نكرر المنطق داخل الصفحات.

وظيفته:

```txt
- من يحق له رؤية ماذا؟
- ما الفصول المسندة للمستخدم؟
- ما الطلاب الذين يظهرون له؟
- ما المعلمين المسندون للمشرف؟
- هل المستخدم يستطيع إدخال قياس لهذا الفصل؟
- هل هذا القياس يحتاج فاقدًا؟
- كيف نحسب مؤشر التحسن؟
- كيف نبني دفعة قياس؟
- كيف نوزع التقييمات؟
- كيف نحسب ملخصات التقارير؟
```

الفصل المقترح:

```txt
packages/contracts = شكل البيانات
packages/domain = منطق التشغيل والحسابات والصلاحيات
```

## 5. الفكرة المركزية: التشغيل اليومي المنظم

أي عملية يومية في المنصة يجب أن تمر بهذا النمط:

```txt
Actor → Scope → Assignment → Template → Batch → Records → Reports
```

### Actor

الشخص الذي ينفذ العملية.

مثال:

```txt
معلم
وكيل
مشرف
مدير
معلمة قيم
مشرف باص
```

### Scope

النطاق الذي يعمل عليه.

مثال:

```txt
مؤسسة
مدرسة
سنة دراسية
صف
فصل
خط باص
مجموعة معلمين
طالب
```

### Assignment

الإسناد الذي يحدد ماذا يرى وماذا يفعل.

مثال:

```txt
معلم مسند لفصل
مشرف مسند لمجموعة معلمين
مشرف باص مسند لخط
وكيل له صلاحية حضور المدرسة
معلمة قيم تغطي فصولًا معينة
```

### Template

القالب أو النموذج.

مثال:

```txt
اختبار فتري أول
قياس مركزي
تقييم قيم - الوحدة الأولى
تقييم أركان - الأسبوع الثاني
تقييم معلم أسبوعي
زيارة مشرف
حضور يومي
```

### Batch

دفعة تشغيل جماعية.

مثال:

```txt
المعلم أدخل قياسًا لفصل كامل
الوكيل سجل حضور فصل كامل
معلمة القيم قيّمت طلاب فصل في وحدة معينة
المدير قيّم مجموعة معلمين في الأسبوع الثالث
```

### Records

السجلات الفردية الناتجة.

مثال:

```txt
درجة طالب في قياس
حضور طالب في يوم
ملاحظة على طالب
تقييم معلم
حضور طالب في الباص
```

### Reports

الملخصات والتقارير.

مثال:

```txt
تقرير حضور طالب
تقرير حضور فصل
تقرير فاقد تعليمي
تقرير قياسات فصل
تقرير تقييم معلم
تقرير أداء مشرف
```

---

## 6. قاعدة مهمة جدًا

لا نحفظ التشغيل الجماعي كسجلات فردية فقط.

بل نحفظ:

```txt
Batch
+
Records فردية
```

مثال القياسات:

```txt
StudentMeasurementBatch
+
StudentAssessmentRecord لكل طالب
```

مثال الحضور:

```txt
StudentAttendanceBatch
+
StudentAttendanceRecord لكل طالب
```

مثال القيم والأركان:

```txt
KGSpecialistMeasurementBatch
+
StudentAssessmentRecord أو StudentSpecialistRecord لكل طالب
```

مثال تقييم المعلمين:

```txt
EvaluationCycle / EvaluationBatch
+
EvaluationSubmission لكل موظف أو معلم
```

## 7. ملف الطالب

ملف الطالب في `web-admin` لا يكون مكان التشغيل الأساسي.

وظيفته:

```txt
- عرض كل ما حدث للطالب
- الحضور
- القياسات
- المتابعات
- الفاقد التعليمي
- الملاحظات
- القضايا
- النقل
- التحفيز
- تاريخ ولي الأمر
```

أي أن:

```txt
ملف الطالب = سجل شامل
وليس واجهة الإدخال اليومية الأساسية
```

---

## 8. ملف الموظف

نحتاج لاحقًا ملف موظف مشابه.

وظيفته:

```txt
- بيانات الموظف
- عضوياته
- أدواره
- إسناداته
- تقييماته
- الزيارات
- الملاحظات الإدارية
- سجلات الأداء
```

---

## 9. Staff Portal حسب الدور

كل مستخدم في `web-staff` يرى واجهة مختلفة حسب:

```txt
- عضويته
- دوره
- نطاقه
- إسناداته
- السياسات المفعلة
```

مثال:

```txt
معلم فصل:
يرى فصوله وطلابه وقياسات الفصل وملاحظات الطلاب.

وكيل شؤون الطلاب:
يرى فصول المدرسة وحضور اليوم والغياب والتأخر والقضايا.

معلمة قيم:
ترى الفصول المسندة لها وتقييمات القيم حسب الوحدة والأسبوع.

مشرف تربوي:
يرى المعلمين المسندين له والتقييمات والزيارات المطلوبة.

مشرف باص:
يرى خطوطه وطلاب الخط وحضور الصعود والنزول.
```

## 10. تقييم المعلمين والإداريين

هذا أيضًا يجب أن يدخل تحت نفس فلسفة التشغيل.

النمط:

```txt
Evaluator
→ Target Staff
→ Evaluation Template
→ Cycle
→ Submission
→ Item Scores
→ Summary
```

أمثلة:

```txt
مدير يقيّم المعلمين أسبوعيًا
وكيل المعلمين يقيّم المعلمين أسبوعيًا
وكيل المعلمين يعمل زيارات فترية
مشرف يقيّم معلمين محددين
مشرفة تقيّم معلمات محددات
إدارة الروضة لها نمط مختلف
```

الاختلاف بين البنين والبنات والروضات لا يكون hardcoded قدر الإمكان، بل من خلال:

```txt
- EvaluationFramework
- EvaluationPlan
- EvaluationCycle
- EvaluatorPolicy
- TargetAssignments
- Membership links
```

## 11. DIY الذكي

المطلوب ليس نظامًا مفتوحًا بلا قيود، بل:

```txt
DIY مضبوط بقوالب وسياسات
```

أي أن الإدارة تستطيع لاحقًا تعريف:

```txt
- قالب قياس جديد
- قالب متابعة جديد
- قالب تقييم موظف
- بنود تقييم
- درجات
- صلاحيات
- دورات
- أسابيع
- زيارات
- اعتماد
```

لكن داخل إطار مضبوط حتى تبقى التقارير ممكنة.

---

## 12. ما نحتاج إعادة التفكير فيه قبل التنفيذ

قبل كتابة كود جديد، نراجع هذه النقاط:

```txt
1. هل نضيف apps/web-staff الآن؟
2. كيف ننظم packages/domain؟
3. هل نضيف Batch schemas للقياسات والحضور والنقل؟
4. هل نضيف itemScores موحدًا؟
5. هل نوسع TeacherAssignment ليغطي تشغيل القيم والأركان أفضل؟
6. هل نحتاج OperationalAssignment عام أم نبدأ بـ TeacherAssignment ونوسع؟
7. كيف نربط تقييم الموظفين بنفس فلسفة التشغيل؟
```

---

## 13. القرار المبدئي

القرار المبدئي المقترح:

```txt
1. نثبت web-admin للإدارة والملفات والتقارير.
2. ننشئ web-staff للتشغيل اليومي.
3. ننشئ packages/domain من البداية.
4. نبقي packages/contracts مصدرًا للسكيما.
5. نعتمد Template / Assignment / Batch / Record كنمط عام.
6. لا نهدم ما تم، بل نعيد توجيهه:
   - صفحات الطالب = قراءة وسجل شامل
   - Staff Portal = إدخال وتشغيل يومي
```

## 14. أول خطوة عملية بعد اعتماد الوثيقة

أقترح أن تكون أول خطوة عملية لاحقًا:

```txt
تصميم packages/domain
```

ثم بعدها:

```txt
تصميم Batch schemas
```

ثم:

```txt
تصميم web-staff كبوابة تشغيل
```

أما اقتراحك بخصوص تجهيز بيانات الموظفين الحقيقية وصلاحياتهم في Google Sheet أو Excel ثم عمل seed، فهو ممتاز، ونرجع له بعد تثبيت هذه الوثيقة.
