## تصميم `web-staff` على الورق

`web-staff` هو بوابة التشغيل اليومي.
هدفه أن يدخل الموظف بإيميله فيرى فقط ما يخصه حسب:

```txt
الدور + العضوية + الإسنادات + الصلاحيات + نطاق المدرسة/الفصل/الخط/الموظفين
```

وليس المطلوب منه أن يتعامل مع شجرة `web-admin`.

---

# 1. فكرة `web-staff`

الواجهة الرئيسية لا تبدأ من “بحث عن طالب”، بل تبدأ من:

```txt
ماذا عليّ اليوم؟
وما النطاقات التي أستطيع العمل عليها؟
```

لذلك الواجهة تكون مبنية حول:

```txt
الرئيسية
فصولي
مهامي
تقييماتي
الطلاب
النقل
القضايا
الملاحظات
```

لكن ظهور هذه المساحات يعتمد على صلاحيات المستخدم.

---

# 2. الرئيسية

المسار المقترح:

```txt
/apps/web-staff
```

أو داخل التطبيق:

```txt
/
```

محتوى الصفحة الرئيسية:

```txt
- ترحيب باسم المستخدم
- المؤسسة/المدرسة الحالية
- مهامي اليوم
- الفصول المسندة لي
- تنبيهات ناقصة
- تقييمات مطلوب تنفيذها
- قضايا محالة لي
- دفعات غير مكتملة
```

مثال:

```txt
اليوم لديك:
- تسجيل حضور فصل 3-أ
- قياس فتري لفصل 2-ب
- تقييم أسبوعي لمعلمَين
- قضية طالب بانتظار إجراء
- دفعة قياس محفوظة كمسودة
```

هذه الصفحة تعتمد على `packages/domain/staff-workspace`.

---

# 3. فصولي

المسار المقترح:

```txt
/classes
```

يعرض الفصول التي يحق للمستخدم العمل عليها.

ليس فقط معلم الفصل؛ قد تظهر الفصول بسبب:

```txt
- TeacherAssignment
- OperationalAssignment
- صلاحية وكيل على مدرسة كاملة
- معلمة قيم تغطي كل فصول الروضة
- معلمة أركان تغطي فصول معينة
- مشرف له نطاق متابعة
```

كل فصل يظهر كبطاقة:

```txt
اسم الفصل
المدرسة
الصف/المستوى
عدد الطلاب
المهام المتاحة
```

داخل الفصل:

```txt
/classes/[classId]
```

تظهر عمليات الفصل:

```txt
- طلاب الفصل
- حضور اليوم
- قياسات ومتابعات
- ملاحظات
- تحفيز
- إحالة قضية
```

لكن حسب صلاحية المستخدم.

مثال:
معلم قد يرى القياسات والملاحظات.
وكيل شؤون الطلاب قد يرى الحضور والقضايا.
معلمة القيم ترى تقييم القيم.
معلمة الأركان ترى تقييم الأركان.

---

# 4. مهامي

المسار المقترح:

```txt
/tasks
```

هذه صفحة تجمع كل المطلوب من المستخدم، بغض النظر عن نوع المهمة.

أنواع المهام:

```txt
STUDENT_ATTENDANCE
STUDENT_MEASUREMENT
STUDENT_TRACKER
KG_VALUES_EVALUATION
KG_CORNERS_EVALUATION
STUDENT_CASE_HANDLING
TRANSPORT_ATTENDANCE
STAFF_EVALUATION
CUSTOM
```

كل مهمة تظهر هكذا:

```txt
العنوان
النوع
النطاق
الموعد أو الأسبوع
الحالة
زر فتح
```

مثال:

```txt
قياس فتري أول - فصل 2-أ
الحالة: لم يبدأ

حضور اليوم - فصل 3-ب
الحالة: مفتوح

تقييم أسبوعي - المعلم أحمد
الحالة: مسودة
```

المهام تُبنى من:

```txt
OperationalAssignment
+
Plans / Cycles
+
Batches
+
Submissions
```

---

# 5. تقييماتي

المسار المقترح:

```txt
/evaluations
```

هذه خاصة بتقييم المعلمين أو الإداريين.

تظهر للمستخدم إذا كان لديه صلاحية تقييم.

الصفحة تعرض:

```txt
- التقييمات المطلوبة
- الدورات المفتوحة
- المعلمون/الموظفون المسندون له
- المسودات
- المكتمل
- الناقص
```

مثال:

```txt
الأسبوع الثالث
- المعلم أحمد: لم يبدأ
- المعلم خالد: مسودة
- المعلم سعيد: مكتمل
```

وتدعم لاحقًا:

```txt
- تقييم أسبوعي
- زيارة فترية
- زيارة مشرف
- تقييم إداري
```

القاعدة التي ثبتناها:

```txt
درجة الأسبوع من 100
والتراكمي = متوسط الأسابيع المفعّلة/المعتمدة فقط
```

---

# 6. قياسات ومتابعات الفصل

داخل الفصل:

```txt
/classes/[classId]/measurements
```

الغرض:

```txt
إدخال نتائج لطلاب الفصل دفعة واحدة
```

التدفق:

```txt
اختيار نوع العملية
اختيار القالب
عرض طلاب الفصل
إدخال النتائج
حفظ مسودة
إرسال
إنشاء سجلات فردية لكل طالب
```

يدعم:

```txt
- قياس رسمي
- متابعة مستمرة
- قيم
- أركان
- قرآن
- فاقد تعليمي
```

وينتج:

```txt
StudentMeasurementBatch
+
StudentAssessmentRecord أو StudentTrackerEntry لكل طالب
```

---

# 7. حضور اليوم

المسار المقترح:

```txt
/classes/[classId]/attendance
```

أو صفحة عامة:

```txt
/attendance
```

الفرق:

```txt
/classes/[classId]/attendance
```

تسجيل حضور فصل محدد.

```txt
/attendance
```

واجهة وكيل أو مسؤول يرى كل الفصول المسموحة له.

التدفق:

```txt
اختيار اليوم
عرض طلاب الفصل
تحديد حاضر / غائب / متأخر / بعذر
حفظ
إرسال
```

لاحقًا ينتج:

```txt
StudentAttendanceBatch
+
StudentAttendanceRecord لكل طالب
```

---

# 8. القضايا والإحالات

المسارات المقترحة:

```txt
/cases
/classes/[classId]/cases
/students/[studentId]/cases/new
```

في `web-staff` لا نعرض إدارة كاملة للقضايا مثل admin، بل نعرض ما يخص المستخدم:

```txt
- إحالة طالب
- القضايا المحالة لي
- الإجراءات المطلوبة مني
- الرد أو التحويل
- الإغلاق إذا كان مصرحًا
```

مثال:

```txt
المعلم يحيل طالبًا للوكيل
الوكيل يستقبل القضية
الموجه الطلابي يضيف إجراء
المدير يراجع إذا احتاج الأمر
```

---

# 9. النقل والباص

المسار المقترح:

```txt
/transport
/routes/[routeId]
```

يظهر فقط لمن له:

```txt
TRANSPORT_ATTENDANCE
```

أو دور مشرف باص/نقل.

التدفق:

```txt
اختيار الخط
عرض طلاب الخط
تسجيل صعود / نزول / غياب
إضافة ملاحظة
حفظ الدفعة
```

ينتج لاحقًا:

```txt
TransportAttendanceBatch
+
StudentTransportAttendanceRecord لكل طالب
```

---

# 10. الملاحظات والتحفيز

الملاحظات تكون سهلة وسريعة.

من داخل الفصل أو الطالب:

```txt
/classes/[classId]/notes
/students/[studentId]/notes/new
```

لكن في `web-staff` الأفضل أن تكون مرتبطة بالسياق:

```txt
من الفصل
اختر الطالب
اكتب الملاحظة
حدد التصنيف والظهور
```

التحفيز مشابه:

```txt
اختر طالبًا أو مجموعة طلاب
اختر نوع التحفيز
احفظ
```

لاحقًا ينتج:

```txt
StudentNote
GamificationEvent
```

---

# 11. مبدأ الصلاحيات في `web-staff`

لا نضع الصلاحيات داخل الصفحات مباشرة.

الصفحة تسأل `domain`:

```txt
ما الذي يراه هذا المستخدم؟
ما العمليات المتاحة له؟
هل يستطيع فتح هذا الفصل؟
هل يستطيع تنفيذ هذا القياس؟
هل يستطيع تقييم هذا الموظف؟
```

مثال:

```txt
domain.buildStaffHome(...)
domain.getVisibleClassesForActor(...)
domain.canRunOperation(...)
domain.buildEvaluationTasksForActor(...)
```


# 13. أول نسخة من `web-staff`

أقترح أن تكون أول نسخة محدودة وليست ضخمة:

```txt
1. تسجيل دخول
2. اختيار المؤسسة
3. الرئيسية
4. فصولي
5. صفحة فصل
6. قياس جماعي لطلاب الفصل
7. مهامي
```

ثم نضيف بعدها:

```txt
8. تقييماتي
9. حضور اليوم
10. القضايا
11. النقل
12. التحفيز
```

# 14. القرار المعتمد مبدئيًا

```txt
web-staff = تشغيل يومي حسب الإسناد
web-admin = إدارة وتقارير وملفات شاملة
packages/domain = منطق الصلاحيات والتشغيل
contracts = شكل البيانات
```

وأول Workflow عملي لاحقًا داخل `web-staff` يكون:

```txt
فصولي
→ اختيار فصل
→ قياس/متابعة جماعية
→ إنشاء StudentMeasurementBatch
→ عند الإرسال إنشاء سجلات فردية
→ تظهر النتائج في ملف الطالب داخل web-admin
```

بهذا نكون رسمنا `web-staff` على الورق بشكل كافٍ كبداية.

فهمت منك أنك تعترض على فكرة أن تكون أول نسخة من `web-staff` محدودة، وترى أن الأفضل أن نخططها من البداية كمنظومة كبيرة ومكتملة نسبيًا، تشمل أغلب مساحات التشغيل اليومي، حتى لو أخذ التخطيط والتنفيذ وقتًا أطول.

يعني بدل:

```txt
نسخة أولى صغيرة:
فصولي + قياسات فقط
```

تريد أن نفكر في:

```txt
web-staff كامل من البداية:
الرئيسية
فصولي
طلابي
الحضور
القياسات والمتابعات
القيم والأركان
الفاقد
الملاحظات
القضايا
التحفيز
النقل والباص
تقييماتي
المعلمون/الموظفون المسندون لي
المهام والتنبيهات
```

مع تصميم معماري يسمح أن كل هذه المساحات تعمل بنفس فلسفة:

```txt
Role + Assignment + Scope + Task + Batch + Record + Report
```

تمام. إذن نعدّل القرار:

```txt
web-staff لا يكون نسخة صغيرة مؤقتة.
بل نخططه من البداية كمنظومة تشغيل يومي شبه مكتملة،
حتى لو نفذناها على مراحل داخل نفس التصميم.
```

## التصور الجديد لـ `web-staff`

`web-staff` يكون التطبيق الرئيسي لكل شخص يعمل يوميًا داخل المدرسة أو المؤسسة، حسب دوره وإسناداته.

يعني يدخل المستخدم فيجد:

```txt
الرئيسية
فصولي
طلابي
الحضور
القياسات والمتابعات
القيم والأركان
الفاقد التعليمي
الملاحظات
القضايا والإحالات
التحفيز
النقل والباص
تقييماتي
الموظفون/المعلمون المسندون لي
المهام والتنبيهات
```

لكن لا تظهر كل المساحات لكل شخص.
الظهور يكون حسب:

```txt
Membership
OperationalAssignment
TeacherAssignment
Evaluation Assignment
School / Class / Route Scope
```

## التقسيم المقترح داخل `web-staff`

### 1. الرئيسية

تعرض ملخص اليوم:

```txt
- مهامي المفتوحة
- دفعات غير مكتملة
- فصولي
- تقييماتي المطلوبة
- قضايا محالة لي
- حضور لم يكتمل
- تنبيهات الفاقد
- مهام النقل إن وجدت
```

هذه الصفحة تعتمد على `packages/domain/staff-workspace`.

---

### 2. فصولي

ليست فقط للمعلم.

تظهر لكل شخص لديه صلاحية تشغيل على فصول، مثل:

```txt
معلم فصل
معلم مادة
وكيل شؤون الطلاب
معلمة قيم
معلمة أركان
مشرف تربوي له نطاق فصول
```

داخل كل فصل:

```txt
- طلاب الفصل
- الحضور
- القياسات والمتابعات
- القيم والأركان
- الملاحظات
- التحفيز
- الإحالات
```

---

### 3. طلابي

تعرض الطلاب الذين يدخلون ضمن نطاق المستخدم.

مثال:

```txt
معلم يرى طلاب فصوله
وكيل يرى طلاب المدرسة
مشرف باص يرى طلاب خطوطه
موجه طلابي يرى الطلاب المحالين له أو ضمن نطاقه
```

هذه ليست بديلًا عن ملف الطالب في `web-admin`، لكنها نسخة تشغيلية مختصرة.

---

### 4. الحضور

مساحة تشغيلية لتسجيل الحضور اليومي.

تدعم:

```txt
- حضور فصل
- حضور مدرسة حسب صلاحية الوكيل
- غياب
- تأخر
- عذر
- تعديل لاحق بصلاحية
- دفعات حضور
```

وتنتج لاحقًا:

```txt
StudentAttendanceBatch
StudentAttendanceRecord
```

---

### 5. القياسات والمتابعات

هذه مساحة الإدخال الجماعي.

تدعم:

```txt
- اختبار تشخيصي
- اختبار فتري
- قياس مركزي
- قياسات الروضة
- متابعة قرآن
- متابعة فاقد
- متابعة مخصصة
```

وتنتج:

```txt
StudentMeasurementBatch
StudentAssessmentRecord
StudentTrackerEntry
```

---

### 6. القيم والأركان

نجعلها مساحة واضحة، لا ندفنها داخل “متابعة مخصصة”.

لأن معلمة القيم ومعلمة الأركان لهما تشغيل مختلف:

```txt
اختيار الفصل
اختيار الوحدة
اختيار الأسبوع
ظهور كل الطلاب
ظهور البنود
إدخال درجات كل طالب في كل بند
حفظ دفعة
إرسال
```

وتستخدم نفس فكرة:

```txt
StudentMeasurementBatch
itemScores
```

---

### 7. الفاقد التعليمي

تظهر فيها:

```txt
- خطط الفاقد المفتوحة
- الطلاب الذين يحتاجون خطة
- القياس الأول للخطة
- القياس الثاني للخطة
- مؤشر التحسن
```

قد تظهر للمعلم أو المشرف أو الوكيل حسب التكليف.

---

### 8. الملاحظات

مساحة لإضافة ملاحظات على طالب أو مجموعة طلاب.

تحتاج لاحقًا سياسة ظهور:

```txt
داخلية فقط
تظهر للإدارة
تظهر لولي الأمر
تظهر للمعلم فقط
```

---

### 9. القضايا والإحالات

تدعم:

```txt
- إحالة طالب
- استقبال قضية محالة
- إضافة إجراء
- تحويل لوكيل أو موجه أو مدير
- إغلاق حسب الصلاحية
```

المعلم غالبًا يبدأ إحالة، والوكيل/الموجه/الإدارة يكملون المسار.

---

### 10. التحفيز

مساحة سهلة وسريعة:

```txt
اختيار طالب أو مجموعة طلاب
اختيار نوع التحفيز
إضافة نقاط أو شارة أو ملاحظة إيجابية
حفظ الحدث
```

وتظهر لاحقًا في ملف الطالب والتقارير.

---

### 11. النقل والباص

تظهر لمن لديه تكليف نقل:

```txt
مشرف باص
مسؤول نقل
وكيل مصرح له
```

وتدعم:

```txt
- خطوطي
- طلاب الخط
- صعود
- نزول
- غياب عن الباص
- ملاحظات النقل
```

---

### 12. تقييماتي

خاصة بتقييم المعلمين والإداريين.

تعرض:

```txt
- الدورات المفتوحة
- الأسابيع المفعلة
- الزيارات المطلوبة
- المعلمون/الموظفون المسندون لي
- المسودات
- المكتمل
- الناقص
```

وتعتمد على منطق التقييم الموجود عندنا، مثل توزيع التقييمات، بناء الملخصات، ومعرفة المستهدفين؛ وهذه الملفات الحالية مكانها الطبيعي لاحقًا داخل `packages/domain` بدل بقائها متفرقة في lib.   

---

### 13. الموظفون/المعلمون المسندون لي

هذه مهمة للمشرف والمدير والوكيل.

تعرض:

```txt
- المعلمون المسندون للمشرف
- الموظفون ضمن نطاق المدير
- المعلمون المطلوب تقييمهم
- سجل مختصر لكل معلم
```

---

## القرار المحدث

بدل أن نقول:

```txt
نبدأ web-staff صغيرًا
```

نقول:

```txt
نصمم web-staff كاملًا من البداية،
لكن ننفذ وحداته على مراحل دون تغيير المعمارية.
```

يعني التصميم شامل من البداية، والتنفيذ تدريجي.

```txt
الرئيسية

الفاقد
القضايا
التقييمات
النقل

```

لكن كل عنصر يظهر أو يختفي حسب الصلاحيات.

## ما نحتاجه قبل تنفيذ `web-staff`

حتى يكون قويًا من البداية، نحتاج تثبيت هذه العناصر في التصميم:

```txt
1. packages/domain
2. OperationalAssignment
```

## Staff Workspace Task Model

هذا النموذج يجيب على سؤال:

```txt
ما المطلوب من هذا الموظف اليوم أو هذا الأسبوع؟
```

ويغذي:

```txt
- الرئيسية
- صفحة مهامي
- التنبيهات
- العدادات
- روابط الدخول السريع
```

# 1. ما هي `StaffTask`؟

هي مهمة تشغيلية تظهر للمستخدم في `web-staff`.

مثال:

```txt
سجل حضور فصل 3-أ اليوم
أدخل قياس فتري أول لطلاب فصل 2-ب
قيّم المعلم خالد في الأسبوع الثالث
راجع خطة فاقد للطالب أحمد
تابع قضية الطالب محمد
سجل حضور خط الباص رقم 1
```

المهمة ليست بالضرورة سجلًا نهائيًا، بل قد تكون:

```txt
- مطلوب تنفيذها
- مسودة
- ناقصة
- متأخرة
- مكتملة
- بانتظار مراجعة
```

# 12. هل نخزن `StaffTask` في Firestore؟

هنا عندنا خياران.

## الخيار الأول: مهام محسوبة فقط

لا نخزن `StaffTask`، بل `domain` يبنيها من البيانات الحالية.

ميزة:

```txt
لا يوجد تكرار.
```

عيب:

```txt
قد يكون الحساب ثقيلًا إذا كبرت البيانات.
```

## الخيار الثاني: نخزن Read Model للمهام

نخزن:

```txt
orgs/{orgId}/staffTasks/{taskId}
```

ميزة:

```txt
سريع للرئيسية والتنبيهات.
```

عيب:

```txt
نحتاج تحديثها عند تغير الدفعات والقضايا والتقييمات.
```

## رأيي

نبدأ بتصميم الاثنين:

```txt
StaffTask = نوع موحد في domain
StaffTaskReadModel = لاحقًا للتخزين السريع
```

في البداية يمكن بناؤها ديناميكيًا.
وعندما تكبر المنصة، نحفظها كـ Read Model عبر functions.

# 16. القرار المقترح

نعتمد `StaffTask` كنموذج موحد لمهام `web-staff`.

```txt
StaffTask = مهمة تشغيلية موحدة تظهر للمستخدم حسب دوره وإسناداته.
```

```txt
الصفحات لا تبحث عشوائيًا عن المطلوب.
```

## Staff Home / الرئيسية في `web-staff`

الرئيسية هي صفحة “ماذا عليّ الآن؟” وليست لوحة إحصاءات فقط.

```txt
تجميع أهم ما يخص الموظف اليوم أو هذا الأسبوع
حسب الدور والإسناد والصلاحيات
```

## 1. أعلى الصفحة

تعرض:

```txt
السلام عليكم، {اسم الموظف}
المؤسسة / المدرسة الحالية
الدور الحالي
تاريخ اليوم
```

وممكن يظهر تبديل المؤسسة إذا كان المستخدم يعمل في أكثر من مؤسسة أو مدرسة.

## 2. كروت الملخص السريعة

كروت صغيرة تعطي صورة سريعة:

```txt

المسودات
التقييمات المطلوبة
القضايا المفتوحة
دفعات غير مكتملة
```

```txt
دفعات قياس غير مكتملة: 2
```

الكروت التي لا تخص المستخدم لا تظهر.

## تصميم مساحة “فصولي” في `web-staff`

“فصولي” ليست للمعلم فقط.
هي مساحة تعرض أي فصل يستطيع الموظف العمل عليه حسب دوره وإسناداته.

يعني قد تظهر لـ:

```txt id="6wlu2h"
- معلم فصل
- معلم مادة
- وكيل شؤون الطلاب
- معلمة قيم
- معلمة أركان
- معلمة قرآن
- مشرف تربوي
- أي دور له تشغيل على فصول
```

---

## 1. الهدف من “فصولي”

تعطي المستخدم مدخلًا سريعًا إلى الفصول التي يستطيع تنفيذ عمليات عليها.

العمليات مثل:

```txt id="59iomj"
- عرض طلاب الفصل
- تسجيل حضور
- إدخال قياس
- إدخال متابعة
- تقييم قيم
- تقييم أركان
- إضافة ملاحظة
- تحفيز الطلاب
- إحالة طالب لقضية
```

---

## 2. مصدر الفصول

الفصول لا تظهر عشوائيًا.

تُبنى من:

```txt id="y1vcx6"
TeacherAssignment
OperationalAssignment
Membership scope
School scope
Role permissions
```

مثال:

```txt id="mzyqvj"
معلم فصل:
يرى الفصول المسندة له.

معلمة قيم:
ترى فصول الروضة التي يشملها تكليف القيم.

وكيل شؤون الطلاب:
يرى فصول المدرسة أو المدارس المصرح بها.

مشرف تربوي:
يرى الفصول المرتبطة بالمعلمين المسندين له، إن كان له هذا النوع من الوصول.
```

هذه القواعد تكون في `packages/domain`.


كل بطاقة فصل تحتوي:

```txt id="b7pklm"

السنة الدراسية

```

## الخطوة التالية

بعد “فصولي”، نكمل بتصميم:

```txt id="b5vf4x"
مهامي
```

لأنها ستجمع كل العمليات المطلوبة من الموظف، سواء كانت مرتبطة بفصل أو طالب أو موظف أو قضية أو خط نقل.

## تصميم مساحة “تقييماتي” في `web-staff`

مساحة **تقييماتي** هي مركز تشغيل تقييم المعلمين والإداريين.
تظهر فقط لمن لديه دور أو تكليف يجعله مقيّمًا، مثل:

```txt
مدير
مديرة
وكيل شؤون المعلمين
وكيلة
مشرف
مشرفة
قائد/ة روضة
عضو لجنة تقييم
أي دور مكلّف بالتقييم لاحقًا
```

---

# 1. الهدف من “تقييماتي”

تجيب على سؤالين:

```txt
ما التقييمات المطلوبة مني؟
وما حالة كل تقييم؟
```

يعني المقيّم لا يبحث عن المعلم يدويًا، بل يرى:

```txt
الأسبوع الحالي
الدورات المفتوحة
المعلمون/الموظفون المطلوب تقييمهم
المسودات
الناقص
المكتمل
المتأخر
```

---

# 2. أنواع التقييمات

المساحة يجب أن تدعم أكثر من نوع:

```txt
تقييم أسبوعي
تقييم فتري
زيارة صفية
زيارة مشرف
تقييم إداري
تقييم لجنة
تقييم مخصص
```

ولا نجعلها hardcoded للبنين أو البنات أو الروضات، بل تعتمد على القوالب والسياسات.

---

# 3. المكونات الأساسية

نستخدم نفس فلسفة التشغيل:

```txt
Evaluator → Targets → Plan → Cycle → Submission → ItemScores → Summary
```

## Evaluator

الشخص الذي يقيّم.

مثال:

```txt
مدير
وكيل
مشرف
عضو لجنة
```

## Targets

المستهدفون بالتقييم.

مثال:

```txt
معلمون
معلمات
إداريون
موظفون محددون
```

## Plan

خطة التقييم.

مثال:

```txt
تقييم أسبوعي للمعلمين
زيارة فترية
تقييم إداري شهري
```

## Cycle

الدورة الزمنية.

مثال:

```txt
الأسبوع الأول
الأسبوع الثاني
الزيارة الأولى
الزيارة الثانية
الفترة الأولى
```

## Submission

سجل تقييم شخص واحد بواسطة مقيم واحد في دورة معينة.

مثال:

```txt
المدير قيّم المعلم أحمد في الأسبوع الثالث
```

## ItemScores

درجات البنود.

مثال:

```txt
التخطيط: 4 من 5
إدارة الصف: 5 من 5
التفاعل مع الطلاب: 4 من 5
```

## Summary

الملخص النهائي.

مثال:

```txt
درجة الأسبوع من 100
متوسط الأسابيع المعتمدة
عدد التقييمات المكتملة
عدد التقييمات الناقصة
```

# 4. صفحة “تقييماتي” الرئيسية

```txt

```

تعرض:

```txt

- المسودات
- المكتمل
- المتأخر
- ملخص سريع
```

```txt
الأسبوع الثالث - تقييم أسبوعي

أحمد محمد: لم يبدأ
خالد صالح: مسودة
ماجد عبدالله: مكتمل
سعيد ناصر: متأخر
```

```txt
4. المتأخر
```

```txt
الأسبوع / الدورة

```

# 8. تقييم اللجنة والأوزان

إذا كان التقييم من أكثر من مقيم، فلا نجعل كل شخص يكتب في نفس السجل.

الأصح:

```txt
كل مقيم له EvaluationSubmission مستقل
ثم نحسب النتيجة النهائية حسب الأوزان
```

مثال:

```txt
المدير: وزن 40%
الوكيل: وزن 30%
المشرف: وزن 20%
عضو اللجنة: وزن 10%
```

كل واحد يعطي تقييمه، ثم النظام يحسب:

```txt
الدرجة النهائية للأسبوع = مجموع الدرجات الموزونة من 100
```

---

# 9. طريقة حساب الأسبوع

ثبتنا قاعدة مهمة:

```txt
لا نقسم على كل أسابيع الدراسة.
نقسم فقط على الأسابيع المفعّلة والمعتمدة.
```

مثال:

```txt
عدد أسابيع الدراسة = 19
الأسابيع المفعّلة والمعتمدة = 10
متوسط المعلم = مجموع درجات هذه الأسابيع ÷ 10
```

وهذا يمنع ظلم المعلم إذا لم تُفتح بعض الأسابيع أصلًا.

# 10. حالات التقييم

```txt
DRAFT
SUBMITTED

RETURNED

```

في الواجهة:


## الفرق بين MISSING و DRAFT

```txt

MISSING:
كان مطلوبًا تقييم هذا الشخص في هذه الدورة، لكن لم يبدأ بعد.
```

## درجة الأسبوع


```txt
المتوسط التراكمي = متوسط الأسابيع المفعّلة والمعتمدة فقط
```

```txt

عدد التقييمات المعادة
```

# 14. ماذا يرى المدير أو المشرف؟

حسب صلاحياته:

```txt

- المسودات الخاصة به
- التقييمات التي أرسلها
- التقييمات المتأخرة عليه
- ملخص المستهدفين لديه
```

ولا يرى تقييمات الآخرين إلا إذا كان عنده صلاحية مراجعة أو إدارة.

# 15. ماذا يرى المعلم المستهدف؟

هذا قد يكون في `web-staff` أو لاحقًا في ملف الموظف.

```txt
متوسطه التراكمي
التقييمات المعتمدة فقط
الملاحظات المسموح بإظهارها
```

لكن لا يرى المسودات أو التقييمات غير المعتمدة إلا إذا قررنا ذلك بسياسة.


## الخطوة التالية

بعد تصميم “تقييماتي”، نحتاج تصميم:

```txt
صفحة "طلابي" أو "الطلاب"
```

لأنها ستخدم البحث السريع والوصول للطالب داخل `web-staff` بدون الدخول إلى `web-admin`.

فهمت منك أنك تقصد تحديدًا نقطة **ماذا يرى المعلم المستهدف من تقييمه**، وأنك ترى أن المعلم داخل `web-staff` يجب أن يجد في صفحته أو الرئيسية، بجانب “فصولي” وأعماله اليومية، مساحة تعرض له تقييماته ودرجاته المعتمدة، بدل أن تكون فقط في `web-admin` أو ملف الموظف.

هل هذا هو قصدك؟


صحيح، هذا أفضل.

نضيف داخل `web-staff` مساحة خاصة للمعلم/الموظف المستهدف تعرض **تقييماته هو**، بجانب مساحات التشغيل مثل “فصولي”.

## الفكرة

في `web-staff` قد يكون نفس الشخص له وضعان:

```txt

- فصولي
- الحضور
- القياسات
- الملاحظات

2. Target
هو نفسه يتم تقييمه:
- تقييم أسبوعي
- زيارة مشرف
- تقييم فتري
- ملاحظات أداء
```

لذلك لا يكفي أن تكون `تقييماتي` للمقيّم فقط. نحتاج التفريق بين:

```txt
تقييمات أُجريها
تقييماتي أنا
```

## التسمية المقترحة في الواجهة

بدل أن تكون كلها تحت “تقييماتي” فقط، الأفضل:

```txt
- تقييمات مطلوبة مني
- تقييماتي الشخصية
```

أو أبسط:

```txt
- أقيّم
- تقييمي
```

لكن عربيًا أوضح:

```txt
تقييماتي المطلوبة
تقييماتي الشخصية
```

## ماذا يرى المعلم في “تقييماتي الشخصية”؟

يرى فقط ما هو مسموح إظهاره حسب السياسة:

```txt
- التقييمات المعتمدة
- الدرجة الأسبوعية من 100
- المتوسط التراكمي
- عدد الأسابيع المعتمدة
- عدد الأسابيع الناقصة إن كان مسموحًا
- الزيارات الصفية المعتمدة
- الملاحظات المسموح بعرضها
```

ولا يرى:

```txt
- المسودات
- تقييمات قيد المراجعة
- تقييمات غير معتمدة
- ملاحظات داخلية للإدارة
```

إلا إذا قررت السياسة غير ذلك.

## داخل الرئيسية

تظهر له بطاقة مختصرة مثل:

```txt
تقييمي هذا الأسبوع
92 / 100

المتوسط التراكمي
88.5 / 100

الأسابيع المعتمدة
8
```

ومع زر:

```txt
عرض تقييماتي
```

## داخل صفحة مستقلة

المسار المقترح:

```txt
/my-evaluations
```

أو:

```txt
/profile/evaluations
```

وأفضلها في `web-staff`:

```txt
/my-evaluations
```

لأنها واضحة للموظف.

## علاقتها بصفحة “تقييماتي” للمقيّم

نقسمها منطقيًا هكذا:

```txt
/evaluations
```

للتقييمات التي يجب على المستخدم تنفيذها تجاه الآخرين.

```txt
/my-evaluations
```

للتقييمات التي تخص المستخدم نفسه كموظف/معلم.

## القرار المحدّث

نضيف إلى تصميم `web-staff`:

```txt
المعلم/الموظف يرى تقييماته الشخصية داخل web-staff
بجوار فصوله ومهامه
لكن لا يرى إلا التقييمات المعتمدة أو المسموح بها حسب السياسة.
```

وهذا يجعل `web-staff` ليس فقط مكان تنفيذ المهام، بل أيضًا مكان متابعة أداء الموظف لنفسه.



## تصميم صفحة “طلابي” في `web-staff`

صفحة **طلابي** ليست بديلًا عن ملف الطالب الكامل في `web-admin`، لكنها صفحة تشغيلية سريعة للموظف.

هدفها:

```txt
الوصول السريع إلى الطلاب الذين يدخلون ضمن نطاق الموظف
حسب الدور والإسناد والصلاحية
```

---

## 1. من يرى صفحة “طلابي”؟

تظهر لأي مستخدم له علاقة تشغيلية بالطلاب، مثل:

```txt
- معلم فصل
- معلم مادة
- وكيل شؤون الطلاب
- معلمة قيم
- معلمة أركان
- معلمة قرآن
- مشرف باص
- موجه / مرشد طلابي
- مشرف تربوي له نطاق طلاب
- أي دور له صلاحية تعامل مع الطلاب
```

---

## 2. مصدر الطلاب

الطلاب لا يظهرون عشوائيًا.

يتم تحديدهم من خلال:

```txt
TeacherAssignment

Membership scope
Class enrollment
Transport route enrollment
Case assignment
Learning loss plan assignment
```

مثال:

```txt
معلم فصل:
يرى طلاب فصوله.

وكيل شؤون الطلاب:
يرى طلاب المدرسة أو المدارس التابعة له.

مشرف باص:
يرى طلاب خطوط الباص المسندة له.

الموجه الطلابي:
يرى الطلاب المحالين له أو الطلاب داخل نطاقه.
```


## 4. ماذا تعرض صفحة “طلابي”؟

```txt

المدرسة
الصف / الفصل

هل لديه قضية مفتوحة؟

```

مثال:

```txt

فاقد مفتوح: نعم
قضية مفتوحة: لا

[عرض مختصر] [ملاحظة] [تحفيز] [إحالة]
```

## 5. الإجراءات السريعة من صفحة الطالب

حسب صلاحية المستخدم تظهر أزرار مثل:

```txt
- عرض البطاقة المختصرة
- إضافة ملاحظة
- تحفيز الطالب
- إحالة قضية
- عرض الحضور
- عرض القياسات
- متابعة خطة فاقد
- ملاحظة نقل
```

لا تظهر كل الأزرار للجميع.

مثال:

```txt
المعلم:
ملاحظة، تحفيز، إحالة، قياسات.

الوكيل:
حضور، قضايا، ملاحظات.

مشرف الباص:
ملاحظة نقل، حضور الباص.

معلمة القيم:
عرض مختصر، تقييم قيم من خلال الفصل لا من الطالب مباشرة.
```

```txt

```

## تصميم مساحة الحضور في `web-staff`

مساحة الحضور هي تشغيل يومي أساسي، وليست مجرد صفحة داخل ملف الطالب.


```txt

- مشرف أو موظف مصرح له
- معلم فصل إذا كانت السياسة تسمح له بتسجيل حضور فصله
```

ولا نفترض أن كل معلم يستطيع تسجيل الحضور إلا إذا أعطيناه تكليفًا أو صلاحية.


```txt

اختيار القالب
عرض الطلاب
إدخال الدرجة أو البنود
حفظ / إرسال
```

وتنتج عند الإرسال:

```txt

```

## 7. علاقة الفاقد التعليمي

في القياسات الرسمية، إذا كان القالب يقول:

```txt

```

وكانت نتيجة الطالب أقل من الحد، يتم تعليم سجل الطالب بأنه يحتاج فاقدًا:

```txt

```

ثم يتم فتح خطة فاقد للطالب من:

```txt

```

أو تظهر لاحقًا في ملف الطالب داخل `web-admin`.

## تصميم الفاقد التعليمي في `web-staff`

مساحة الفاقد التعليمي في `web-staff` ليست لإدارة القوالب، بل للتشغيل والمتابعة اليومية.

هدفها:

```txt
اكتشاف الطلاب المحتاجين لدعم
فتح خطط علاجية
متابعة القياس الأول والثاني
حساب التحسن
إغلاق الخطة أو استمرارها
```

---

## 1. من يستخدم مساحة الفاقد؟

حسب التكليف والصلاحية، قد تظهر لـ:

```txt
- معلم الفصل
- معلم المادة
- المشرف
- وكيل الشؤون التعليمية
- الموجه أو المرشد حسب السياسة
- الإدارة عند المراجعة
```

ولا نثبت شخصًا واحدًا دائمًا؛ المسؤول يحدد من خلال:

```txt
OperationalAssignment
TeacherAssignment
Template policy
Learning loss plan owner
Role permissions
```

---

## 2. مصادر الفاقد

الفاقد قد يأتي من أكثر من مصدر:

```txt
1. قياس رسمي
   مثل فتري أو مركزي أو قياس روضة.

2. متابعة مستمرة
   مثل متابعة قراءة أو قرآن أو مهارة.

3. إدخال يدوي
   إذا لاحظ المعلم أو المشرف وجود فاقد.

4. تقرير أو سياسة لاحقة
   مثل طالب أقل من حد معين خلال فترة.
```

لكن البداية الأقوى:

```txt
قياس رسمي أقل من حد القالب
→ يحتاج خطة فاقد
```

## 3. المسارات المقترحة

داخل `web-staff`:

```txt
/learning-loss/new
/learning-loss/plans/[planId]

/classes/[classId]/learning-loss
```

```txt
/learning-loss
كل خطط الفاقد والطلاب المحتاجون للمتابعة داخل نطاق المستخدم.

إنشاء خطة فاقد يدويًا أو من قياس.

/learning-loss/plans/[planId]
إدارة خطة فاقد محددة.

/classes/[classId]/learning-loss
خطط الفاقد الخاصة بطلاب فصل معين.
```

## 5. “يحتاجون خطة فاقد”


```txt
اختبار فتري أول
42 / 100
أقل من حد القالب 60
[فتح خطة فاقد]
```

## 8. القياس الأول والثاني

```txt
تاريخ القياس الثاني
ملاحظة القياس الثاني
```

```txt
- تحسن جزئي
- لا يوجد تحسن
- تراجع
```

## تصميم الملاحظات في `web-staff`

مساحة الملاحظات ستكون من أكثر المساحات استخدامًا؛ لأنها ليست خاصة بدور واحد.
المعلم، الوكيل، الموجه، المشرف، مشرف الباص، الإدارة… كلهم قد يحتاجون إضافة ملاحظة على طالب أو مجموعة طلاب.

الفكرة:

```txt
الملاحظة = تسجيل واقعة أو ملاحظة تربوية/تعليمية/إدارية/نقلية
لا تصل تلقائيًا إلى قضية أو فاقد إلا إذا قرر المستخدم أو السياسة ذلك
```

## 1. من يستطيع إضافة ملاحظة؟

حسب الصلاحية والتكليف:

```txt
- معلم الفصل
- معلم المادة
- وكيل شؤون الطلاب
- الموجه / المرشد الطلابي
- المشرف التربوي
- معلمة القيم
- معلمة الأركان
- مشرف الباص
- الإدارة
- أي دور مصرح له لاحقًا
```

الصلاحية تأتي من:

```txt
Membership
OperationalAssignment
TeacherAssignment
Role permissions
School / Class / Route scope
```

## 2. أنواع الملاحظات

نحتاج تصنيفًا واضحًا:

```txt
ACADEMIC
تعليمية / أكاديمية

BEHAVIORAL
سلوكية

ADMINISTRATIVE
إدارية

ATTENDANCE
مرتبطة بالحضور

TRANSPORT
مرتبطة بالنقل

LEARNING_LOSS
مرتبطة بالفاقد

POSITIVE
إيجابية / تحفيزية

GENERAL
عامة

CUSTOM
مخصصة
```

مثال:

```txt
ملاحظة تعليمية:
الطالب يحتاج تدريبًا إضافيًا على القراءة.

ملاحظة سلوكية:
تكرر مقاطعة الزملاء أثناء النشاط.

ملاحظة نقل:
لم يلتزم الطالب بالجلوس في المقعد المخصص.

ملاحظة إيجابية:
شارك الطالب بفاعلية في النشاط الجماعي.
```

## 5. سياسة الظهور

هذه أهم نقطة.

نحتاج `visibility` واضحًا:

```txt
PRIVATE_TO_AUTHOR
خاصة بكاتب الملاحظة

STAFF_INTERNAL
داخلية للموظفين المصرح لهم

ADMIN_ONLY
للإدارة فقط

STUDENT_SUPPORT_TEAM
للموجه/الوكيل/فريق الدعم

PARENT_VISIBLE
تظهر لولي الأمر

TRANSPORT_TEAM
لفريق النقل فقط
```

الافتراضي الآمن:

```txt
STAFF_INTERNAL
```

ولا تظهر لولي الأمر إلا إذا اختار المستخدم المصرح أو السياسة ذلك.

```txt
داخلية للموظفين
```

الأفضل ألا نجعل كل تحفيز يظهر تلقائيًا لولي الأمر أو الشاشة الصفية.

## تصميم النقل والباص في `web-staff`

مساحة النقل والباص هي تشغيل يومي مستقل، لأنها لا تعتمد فقط على الفصول، بل على:

```txt
خطوط النقل
طلاب الخط
مشرف الباص
الصعود والنزول
ملاحظات النقل
```

الفكرة:

```txt
مشرف الباص أو مسؤول النقل يدخل
← يرى الخطوط المسندة له
← يفتح خطًا
← يرى طلاب الخط
← يسجل الصعود أو النزول أو الغياب
← يضيف ملاحظات نقل عند الحاجة
← تحفظ السجلات وتظهر لاحقًا في ملف الطالب والتقارير
```

```txt
- مشرفة الباص
- مسؤول النقل
- وكيل/وكيلة مصرح له
- مدير/مديرة عند المراجعة
- الإدارة في التقارير
```

الصلاحية تأتي من:

```txt

Membership
TransportRoute assignment
Role permissions
School scope
Route scope
```

## 10. طريقة تسجيل الصعود والنزول


```txt

وعند الإرسال نثبت السجلات الفردية
```

ولو احتجنا لاحقًا تتبعًا لحظيًا، نضيف سجل أحداث.
