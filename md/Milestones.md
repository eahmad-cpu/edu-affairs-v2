# Milestone 11 — الواجبات وبنك الأسئلة

### 11A — تثبيت التصور المعماري للواجبات وبنك الأسئلة

## الهدف

تحديد الشكل النهائي للعلاقة بين:

```txt
المادة
الفصل الدراسي
بنك الأسئلة
الواجب
أسئلة الواجب
إجابات الطلاب
```

بحيث لا نبني واجبات عشوائية، بل نبنيها ضمن فلسفة التشغيل اليومي:

```txt
Actor → Scope → Assignment → Template/Bank → Batch/Assignment → Records/Submissions
```

وهذا متسق مع تصور `web-staff` كتطبيق تشغيل يومي حسب الدور والإسناد، وليس مجرد صفحات منفصلة. 

## المخرجات

* قرار معتمد لمسار Firestore.
* قرار معتمد للفرق بين:

  * `QuestionBankItem`
  * `StudentHomeworkAssignment`
  * `HomeworkQuestionSnapshot`
  * `StudentHomeworkSubmission`
* تحديد أن السؤال ينشأ مرة في بنك الأسئلة، ثم يؤخذ منه Snapshot داخل الواجب.
* تحديد أن الفصل الدراسي جزء أساسي من السياق.

## يكتمل عندما

* نعتمد أن بنك الأسئلة جزء أساسي من الـ Milestone وليس لاحقًا.
* نعتمد أن الواجب لا يخزن أسئلة عشوائية فقط، بل يعتمد على بنك أسئلة.
* نعتمد أن كل واجب يحمل:

  * `schoolId`
  * `academicYearId`
  * `termId`
  * `gradeId`
  * `classId`
  * `subjectKey`
  * `classSubjectOfferingId`

## الهدف

```txt

```


## يكتمل عندما

* يبني `@takween/contracts` بدون أخطاء.
* تكون السكيما قادرة على تمثيل:

  * سؤال اختيار من متعدد.
  * سؤال صح/خطأ.
  * الإجابة الصحيحة.
  * المشتتات/الاختيارات الخاطئة.
  * ربط السؤال بالصف والمادة والفصل الدراسي.
* تكون السكيما ملتزمة بالقاعدة التي ثبتناها: السياق التشغيلي يجب أن يحمل المدرسة والسنة والفصل الدراسي والمادة بوضوح.

### 11H — إنشاء سؤال أثناء إنشاء الواجب

## الهدف

عدم إجبار المعلم على الذهاب إلى بنك الأسئلة أولًا.

أثناء إنشاء الواجب يستطيع الضغط على:

```txt
+ سؤال جديد
```

فيُنشئ السؤال، ويحفظه النظام تلقائيًا في بنك الأسئلة، ثم يضيف Snapshot منه داخل الواجب.

## المخرجات

داخل صفحة إنشاء الواجب:

```txt
إضافة سؤال جديد
اختيار نوع السؤال
تحديد الإجابة الصحيحة
حفظ السؤال في البنك
إضافته للواجب
```

## يكتمل عندما

* ينشئ المعلم سؤالًا من داخل صفحة الواجب.
* يظهر السؤال مباشرة ضمن أسئلة الواجب.
* يُحفظ السؤال أيضًا في بنك الأسئلة.
* لا يتكرر السؤال إذا حفظ المعلم الواجب أكثر من مرة.

### 11I — نشر الواجب

## الهدف

إتاحة دورة حياة واضحة للواجب.

## المخرجات

حالات الواجب:

```txt
DRAFT
PUBLISHED
CLOSED
CANCELLED
```

والانتقال الأساسي:

```txt
DRAFT → PUBLISHED
```

## يكتمل عندما

* يستطيع المعلم حفظ واجب كمسودة.
* يستطيع نشر الواجب.
* لا يمكن للطالب رؤية الواجب إلا إذا كان `PUBLISHED`.
* لا يمكن تعديل الأسئلة بعد النشر إلا بسياسة واضحة، أو يتم إنشاء نسخة جديدة.

### 11J — صفحة تفاصيل الواجب

## الهدف

عرض الواجب بعد إنشائه، مع أسئلته وسياقه وحالته.

## المخرجات

صفحة مثل:

```txt
/staff/homework/[assignmentId]
```

وتعرض:

```txt
عنوان الواجب
المادة
الفصل
الفصل الدراسي
الحالة
الأسئلة
الطلاب المستهدفين
تاريخ الإتاحة
تاريخ التسليم
```

## يكتمل عندما

* يمكن فتح أي واجب من قائمة الواجبات.
* تظهر الأسئلة كما كانت وقت إنشاء الواجب، لا كما صارت لاحقًا في بنك الأسئلة.
* يظهر السياق كاملًا بوضوح.

### 11K — تجهيز تسليمات الطلاب بدون بناء واجهة الطالب الآن

## الهدف

تجهيز السكيما والمنطق لاستقبال إجابات الطلاب لاحقًا، حتى لو لم نبنِ واجهة الطالب الآن.

## المخرجات

تصميم:

```txt
studentHomeworkAssignments/{assignmentId}/submissions/{studentId}
```

أو Read Model مكافئ لاحقًا.

ويحتوي:

```txt
studentId
answers
score
maxScore
status
startedAt
submittedAt
gradedAt
```

## يكتمل عندما

* نستطيع نظريًا حفظ إجابة طالب.
* نستطيع حساب الدرجة تلقائيًا لأسئلة:

  * اختيار من متعدد.
  * صح/خطأ.
* لا نحتاج إعادة تصميم السكيما عند بناء تطبيق الطالب/ولي الأمر.

### 11L — ربط الواجبات بصفحة المادة والفصل

## الهدف

إظهار الواجبات كجزء طبيعي من سياق المادة، بجانب القياسات وبنك الأسئلة.

## المخرجات

داخل كارت المادة تظهر أزرار مثل:

```txt
إدخال قياس / اختبار
بنك الأسئلة
الواجبات
إنشاء واجب
```

## يكتمل عندما

* يظهر زر “بنك الأسئلة” في كارت المادة.
* يظهر زر “الواجبات” في كارت المادة.
* يظهر زر “إنشاء واجب” في كارت المادة.
* كل الروابط تمرر:

  * `schoolId`
  * `academicYearId`
  * `classSubjectOfferingId`
  * `subjectKey`

### 11M — فحص Firestore وGuardrails

## الهدف

التأكد أن بناء الواجبات وبنك الأسئلة لا يخالف القواعد التي تعلمناها سابقًا.

## المخرجات

Checklist فحص بعد التنفيذ:

```txt
لا نعتمد على classId وحده
السؤال يعرف gradeId + termId + subjectKey
الواجب يعرف schoolId + academicYearId + termId + classId + subjectKey
السؤال داخل الواجب Snapshot وليس reference فقط
seed يدعم dry-run إن وجد
الروضة والابتدائي لا يختلطان
```

## يكتمل عندما

* يتم إنشاء سؤال في بنك الأسئلة.
* يتم إنشاء واجب من السؤال.
* يتم فتح الواجب من صفحة المادة الصحيحة.
* لا تظهر أسئلة مادة في مادة أخرى.
* لا تظهر أسئلة فصل دراسي في فصل دراسي آخر إلا لو كانت صالحة له.
* لا يحدث خلط بين المدارس المتشابهة في `classId`.
